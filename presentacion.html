El problema y el dataset

¿Por qué es relevante?

Las plataformas de e-commerce (MercadoLibre, eBay, etc.) prohíben que los vendedores incluyan datos de contacto en sus publicaciones para evitar que se salten la plataforma. Sin embargo, muchos vendedores esquivan el filtro de texto embebiendo teléfonos, emails o handles de WhatsApp directamente dentro de la imagen.

¿Qué queremos resolver?

Construir un sistema automático que, dada una imagen de publicación, detecte si contienen datos de contacto.

Ejemplo del problema
Imagen con teléfono y WhatsApp incrustados sobre el producto, ilegibles para los filtros de texto plano.

Pipeline YOLO + OCR — Detección entrenada

Por qué lo probamos

Para localizar Y clasificar regiones de texto sospechoso en una sola pasada. Hipótesis: si el detector acota la región, podemos correr OCR únicamente sobre las cajas detectadas (10–100× menos área que OCR full-image), bajando latencia sin perder cobertura. A diferencia de los otros modelos, YOLO es un pipeline multi-etapa, no un solo forward.

Características

  • 4 clases: phone email social keyword
  • Input: imagen 640×640 normalizada
  • Output: lista de bboxes + clase + confidence
  • Entrenado con weak supervision — sin bboxes manuales
  • Variantes probadas YOLOv8n · YOLOv8s

Pipeline (5 etapas)

  1. Pseudo-labels: EasyOCR sobre positivos + regex (phone/email/social/keyword) → bboxes con clase
  2. Formato YOLO: conversión a (cx, cy, w, h) normalizadas + split train/val 85/15 + data.yaml
  3. Entrenamiento: YOLOv8 desde pesos COCO, 50 epochs, batch 16, imgsz 640
  4. Inferencia: detector → bboxes con clase y confidence
  5. Post-proceso: crop por bbox + EasyOCR sobre el crop + regex valida el texto extraído

EfficientNet-B0 — Clasificador binario

Por qué lo probamos

Porque se alinea directamente con los labels que tenemos (imagen-level 0/1), sin necesidad de generar pseudo-anotaciones. Es la baseline supervisada más simple y la que mejor explota lo que el dataset realmente ofrece. Si funciona bien, no necesitamos sumar complejidad.

Características

  • Backbone EfficientNet-B0 pre-entrenado en ImageNet
  • Input: imagen 384×384 normalizada con stats ImageNet
  • Output: probabilidad escalar p(contacto) ∈ [0, 1]
  • Loss: BCEWithLogitsLoss con pos_weight suavizado (√(neg/pos)) para el desbalance
  • Threshold de decisión calibrado en validación interna

Pipeline (4 etapas)

  1. Preprocesamiento: resize 384×384 + normalización ImageNet + augmentations (flip, rotation, color jitter)
  2. Forward: backbone EfficientNet-B0 extrae features de la imagen completa
  3. Cabeza binaria: Linear(in_features, 1) → logit → sigmoid → p(contacto)
  4. Decisión: comparación con threshold calibrado → pred binario

ResNet50 + CAM — Clasificador con localización implícita

Por qué lo probamos

Para obtener bounding boxes "gratis" sin anotación manual: combina un clasificador binario robusto con Class Activation Maps que muestran qué región activó la decisión. Es un punto medio entre clasificación pura (sin localización) y detección entrenada (que requiere pseudo-labels).

Características

  • Backbone ResNet50 pre-entrenado en ImageNet
  • Input: imagen 224×224 normalizada con stats ImageNet
  • Output: pred binario + heatmap CAM 7×7 + bbox derivado + texto OCR
  • Loss: CrossEntropyLoss con class weights inversamente proporcionales
  • OCR de validación: Tesseract sobre la región derivada del CAM

Pipeline (5 etapas)

  1. Clasificación: ResNet50 predice contact / no contact (cabeza FC reemplazada por 2 clases)
  2. Extracción CAM: hook sobre layer4 captura feature maps 7×7 × 2048 + pesos del classifier
  3. Bbox del CAM: heatmap se umbraliza, se upsample a 224×224 y se deriva la región rectangular
  4. OCR sobre la región: Tesseract extrae el texto de la zona localizada
  5. Validación regex: confirma si el texto matchea un patrón de contacto real

Input → Output

Mismo input (test_001049.jpg): clasifica con confianza máxima. El heatmap CAM se omite por falta de checkpoint local.

OCR + Regex — Baseline sin entrenamiento

Por qué lo probamos

Para responder la pregunta más crítica del proyecto: ¿hace falta entrenar algo? Si aplicando OCR + reglas llegamos a métricas competitivas, los modelos entrenados pierden justificación. Es la cota inferior de complejidad contra la que cualquier modelo aprendido debería ganar.

Características

  • EasyOCR con idiomas español + inglés
  • Regex de dominio: teléfonos AR, emails, handles sociales, URLs (IG/FB/WA), keywords ("whatsapp", "contactanos")
  • Sin parámetros aprendidos → cero entrenamiento, fácil de auditar
  • Decisión binaria: positiva si al menos una región matchea algún regex

Pipeline (4 etapas)

  1. OCR full-image: EasyOCR procesa la imagen completa → lista de (bbox, texto, confidence) por región detectada
  2. Filtro por confianza: se descartan regiones con conf < 0.3 (típicas alucinaciones del OCR)
  3. Matching regex: cada texto se evalúa contra los patrones con prioridad email > social > keyword > phone
  4. Agregación: pred = 1 si al menos un match; score = max conf entre regiones que matchearon

Por qué elegimos EfficientNet-B0

Cómo funciona

  1. Backbone EfficientNet-B0 pre-entrenado en ImageNet extrae features visuales de la imagen completa.
  2. Cabeza binaria produce un logit para la clase contacto / no contacto.
  3. Sigmoid convierte el logit en probabilidad de contacto.
  4. Threshold calibrado en validación interna decide si la publicación debe ser marcada para revisión.

Criterio de selección

Como el test set tiene solo 93 positivos sobre 3.000 imágenes, accuracy y F1 aislados pueden ser engañosos. Para el análisis priorizamos recall, AUC-PR y AUC-ROC: métricas más informativas cuando la clase positiva es escasa y perder contactos reales es costoso.

Curvas PR/ROC exportadas desde comparison.ipynb.

Justificación frente a los otros enfoques

vs. OCR full-image + regex
OCR+regex tiene mayor precisión y el F1 puntual más alto, pero detecta solo 18 de 93 positivos. EfficientNet-B0 detecta 70 de 93, por eso es mejor para no dejar pasar publicaciones con contacto.
vs. ResNet50 + CAM
Ambos recuperan 70 positivos, pero EfficientNet-B0 logra menos falsos positivos, mayor precisión, mayor F1 y el mejor AUC-PR.
vs. YOLOv8n / YOLOv8s
YOLO aporta localización, pero a nivel image-level queda por debajo: recall 0.333 y AUC-PR cercano a 0.14. Las pseudo-bboxes generadas con OCR introducen ruido y limitan la performance final.
Modelo final elegido: EfficientNet-B0
Mejor ranking global: AUC-PR 0.208, AUC-ROC 0.880 y recall 0.753 sobre el mismo test set.

Resultados comparativos y elección final

Resumen

Modelo seleccionado
EfficientNet-B0 logra el mejor AUC-PR y AUC-ROC, con el mayor recall junto con ResNet50+CAM.
Baseline más precisa
OCR+regex casi no acusa falsos positivos, pero su recall es demasiado bajo: deja sin detectar 75 de 93 positivos.
YOLO como línea futura
YOLO es útil si la prioridad es localizar la región, pero necesita mejores anotaciones o pseudo-labels menos ruidosas para ganar en test image-level.
